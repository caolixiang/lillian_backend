- Install the coding memo workflow and start executing `plan.md` for the wallet and entitlement backend.
- Establish an isolated worktree before implementation on `main`.

## Scope

- In: Workflow docs, implementation plan tracking, worktree setup, Phase 1 wallet API work, Phase 2 wallet redemption, Phase 3 wallet generation accounting, Phase 4 admin wallet lookup, Phase 5 frontend integration support, wallet task performance optimization, worker concurrency configuration docs, pgxpool connection tuning, activation-token cleanup.
- Out: User-side SPA code changes because this repository only contains the backend and admin frontend.

## Phase Plan

### Phase 0 - Workflow And Worktree Baseline

- Goal: Commit workflow/plan docs and create an isolated implementation worktree.
- Changes: Add memo workflow files, commit `plan.md`, ignore `.worktrees`, create `wallet-entitlements` branch worktree.
- Acceptance: Docs committed on `main`; `.worktrees` ignored; baseline tests run in worktree.
- Status: done

### Phase 1 - Schema And Wallet APIs

- Goal: Add wallet persistence and create/restore/get APIs.
- Changes: Add migrations, recovery code hashing, wallet service/store/API handlers, tests.
- Acceptance: Can create wallet with `0x...` address and one-time recovery code, restore by recovery code, fetch empty entitlements, and verify no plaintext recovery storage.
- Status: done

### Phase 2 - Redemption Into Wallet

- Goal: Redeem `LIL-...` codes into wallet service entitlements and expose service/redeemed wallet fields in admin.
- Changes: Add license redemption columns, implement `/api/wallets/redeem`, update admin license generation/listing for service code and redeemed wallet.
- Acceptance: Admin can create service-specific `image-2-sd`/`image-2-hd` codes; wallet can redeem a code once; wallet entitlements increase correctly.
- Status: done

### Phase 3 - Generation Accounting

- Goal: Bill image generation against wallet service entitlements with reserve/charge/release semantics.
- Changes: Accept `walletAddress`, select service by size tier, reserve wallet entitlement credit/concurrency, mark charged on success, release on failure.
- Acceptance: 1K uses `image-2-sd` first with HD fallback, 2K/4K requires `image-2-hd`, failed generations release reserved credits.
- Status: done

### Phase 4 - Admin Wallet View

- Goal: Add admin wallet lookup with balances, redemption history, and recent task inspection.
- Changes: Add `/admin/wallets/{address}` admin API, Postgres summary queries, admin wallet tab/search UI.
- Acceptance: Admin can inspect a wallet and understand its balances and usage history.
- Status: done

### Phase 5 - Frontend Integration Support

- Goal: Confirm wallet API response shapes and stabilize fields needed by the external SPA wallet modal/task polling.
- Changes: Add task response wallet snapshot and accounting fields; document current wallet/task frontend contract in README.
- Acceptance: External SPA can create/restore wallet, redeem codes, refresh balances, and generate/poll tasks with wallet billing fields.
- Status: done

### Phase 6 - Wallet Task Performance

- Goal: Optimize the wallet task hot paths without adding a new cache dependency.
- Changes: Add targeted Postgres indexes, reduce wallet task polling reads, and let configured worker concurrency control DB pressure.
- Acceptance: Queued/running/wallet task queries have supporting indexes; wallet task lookup filters by `id` and `wallet_id` in SQL; worker startup respects `TaskWorkerConcurrency`; Redis remains deferred.
- Status: done

### Phase 7 - Worker Concurrency Configuration Docs

- Goal: Make `TASK_WORKER_CONCURRENCY` discoverable in env examples and deployment docs.
- Changes: Add the env var to local/production examples and document 8 vCPU / 8 GB VPS tuning guidance.
- Acceptance: Operators can find the startup worker-pool setting and understand it is separate from `/admin` runtime image concurrency.
- Status: done

### Phase 8 - Postgres Pool Configuration

- Goal: Expose only the useful pgxpool knobs for deployment tuning.
- Changes: Add `DB_POOL_MAX_CONNS` and `DB_POOL_MIN_CONNS` to config loading, pgxpool setup, local/production env files, and docs.
- Acceptance: Backend applies explicit pgxpool max/min connections; production guidance stays small and recommends `32` max / `4` min for the 8 vCPU / 8 GB VPS plus Railway Postgres deployment.
- Status: done

### Phase 9 - Remove Legacy Activation Token Flow

- Goal: Make wallet the only user entitlement and generation accounting model.
- Changes: Remove `/api/keys/activate`, `/api/me/credits`, activation verification, direct license credit accounting, legacy task/license columns, and admin UI fallback fields.
- Acceptance: New schema starts without activation/credit ledger columns; existing deployments drop legacy activation tables/columns through migration; admin exchange-code UI uses only `serviceCode` and `credits`; tests and admin build pass.
- Status: done

### Phase 10 - Reset Local Database And Verify Migrations

- Goal: Recreate the local Postgres schema from the backend `.env` `DATABASE_URL` with the current wallet-only migrations.
- Changes: Fix the wallet redemption migration backfill order so legacy HD rows are mapped before defaults are applied, then clear the public schema and rerun all migrations.
- Acceptance: The `.env` database resets cleanly; all migrations apply; legacy activation tables/columns are absent; backend/admin/frontend checks pass.
- Status: done

### Phase 11 - Clarify Admin License Availability

- Goal: Make the admin exchange-code status icon represent availability/expiry only, not whether the code has already been redeemed.
- Changes: Stop treating redeemed wallet fields as an invalid availability state in the license list status icon.
- Acceptance: Redeemed but otherwise active/unexpired exchange codes show the valid check icon, while disabled or expired codes still show the invalid cross; admin build/tests pass.
- Status: done
